Expand description

任务管理器

负责维护系统中所有就绪态任务的队列，提供高效的任务调度服务。 采用 FIFO 调度策略，确保任务调度的公平性和可预测性。

§设计特点

§数据结构

  • 就绪队列: 使用 VecDeque 实现双端队列，支持 O(1) 的队首删除和队尾插入
  • 任务引用: 存储 Arc<TaskControlBlock> 而非任务本身，支持共享所有权
  • 内存布局: 连续内存分配提供良好的缓存局部性

§调度算法

Schedule Order: Task1 → Task2 → Task3 → Task4
         ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐
Enqueue ►│Task4│◄─│Task3│◄─│Task2│◄─│Task1│◄─── Dequeue
         └─────┘  └─────┘  └─────┘  └─────┘
        (最新)                      (最老)

§线程安全

虽然 TaskManager 本身不提供线程安全保证，但通过全局的 UPSafeCell 包装器确保在单处理器环境下的并发安全访问。

§使用模式

通常不直接实例化 TaskManager，而是通过全局函数接口使用：

// 推荐方式：通过全局接口使用
add_task(new_task);
let next_task = fetch_task();

// 不推荐：直接操作实例
let mut manager = TaskManager::new();
manager.add(task);

§性能保证

  • 入队延迟: O(1) 常数时间
  • 出队延迟: O(1) 常数时间
  • 内存开销: 8 bytes per task (仅存储 Arc 指针)
  • 扩容策略: 按需自动扩容，避免频繁内存分配

Fields§

§ready_queue: VecDeque<Arc<TaskControlBlock>>

就绪任务队列

使用双端队列存储所有处于就绪状态的任务控制块。 队列头部是最早加入的任务（下次将被调度）， 队列尾部是最新加入的任务。

§队列特性

  • FIFO 顺序: 先进先出，保证调度公平性
  • 动态大小: 根据系统负载自动调整容量
  • 高效操作: 队首删除和队尾插入均为 O(1)

pub fn new() -> Self

创建新的任务管理器实例

初始化一个空的就绪队列，准备接收和调度任务。 新创建的管理器不包含任何任务，队列容量为 0。

§返回值

返回一个新的 TaskManager 实例，包含：

  • 空的就绪队列（容量为 0）
  • 所有计数器重置为 0
§性能特征
  • 时间复杂度: O(1) - 常数时间初始化
  • 内存分配: 不进行堆内存分配
  • 初始容量: 0，按需扩容以优化内存使用
§使用说明

通常不需要直接调用此函数，而是通过全局 TASK_MANAGER 实例使用：

// 推荐：使用全局实例
use crate::task::manager::{add_task, fetch_task};

// 不推荐：直接创建实例
let manager = TaskManager::new();
§线程安全

new() 函数本身是线程安全的，但返回的实例需要通过 UPSafeCell 等同步原语保护才能在多线程环境中安全使用。

pub fn add(&mut self, task: Arc<TaskControlBlock>)

向就绪队列添加任务

将指定的任务控制块添加到就绪队列的尾部，使其可以被调度器选中执行。 新添加的任务将按照 FIFO 顺序等待调度。

§参数
  • task - 要添加到队列的任务控制块，包装在 Arc 中支持共享所有权
§调度策略

任务按照加入队列的顺序被调度：

调度顺序 (先进先出):

add(Task1) ──► add(Task2) ──► add(Task3)
     │             │             │
     ▼             ▼             ▼
┌─────────┐   ┌─────────┐   ┌─────────┐
│ Task1   │◄──│ Task2   │◄──│ Task3   │
│ (First) │   │(Second) │   │ (Last)  │
└─────────┘   └─────────┘   └─────────┘
     ▲                           ▲
     │                           │
  fetch()                     最新添加
§性能特征
  • 时间复杂度: O(1) 摊还常数时间
  • 内存分配: 队列满时可能触发 O(n) 的扩容操作
  • 扩容策略: 通常按 2 倍容量扩展
§任务状态要求

添加到就绪队列的任务通常应该满足：

  • 任务状态为 TaskStatus::Ready
  • 拥有有效的内存地址空间
  • 陷阱上下文已正确初始化
§使用示例
use alloc::sync::Arc;
use crate::task::task::TaskControlBlock;

let task = Arc::new(TaskControlBlock::new(app_data));
manager.add(task);

// 任务现在在就绪队列中等待调度
println!("任务已添加到就绪队列");
§并发考虑

在并发环境中，应通过适当的同步机制保护此操作：

// 全局接口已提供同步保护
add_task(task); // 推荐方式

// 直接使用需要手动同步
TASK_MANAGER.exclusive_access().add(task);

pub fn fetch(&mut self) -> Option<Arc<TaskControlBlock>>

从就绪队列获取下一个待调度任务

从队列头部移除并返回最早加入的任务，实现 FIFO 调度策略。 如果队列为空，返回 None。

§返回值
  • Some(Arc<TaskControlBlock>) - 成功获取到任务控制块
  • None - 就绪队列为空，没有可调度的任务
§调度行为

按照任务加入队列的顺序进行调度：

队列状态变化:

调用前:
┌─────────┐   ┌─────────┐   ┌─────────┐
│ Task1   │◄──│ Task2   │◄──│ Task3   │
│(Oldest) │   │         │   │(Latest) │
└─────────┘   └─────────┘   └─────────┘
     ▲
  fetch() 返回 Task1

调用后:
┌────---─────┐   ┌─────────┐
│   Task2    │◄──│ Task3   │
│(Now Oldest)│   │(Latest) │
└───────---──┘   └─────────┘
§性能特征
  • 时间复杂度: O(1) 常数时间
  • 内存操作: 仅移动指针，不涉及数据拷贝
  • 缓存效率: 访问队列头部具有良好的缓存局部性
§空队列处理

当队列为空时安全返回 None，调用者应适当处理：

match manager.fetch() {
    Some(task) => {
        println!("获取到任务 PID: {}", task.getpid());
        // 执行任务调度...
    }
    None => {
        println!("没有就绪任务，CPU 进入空闲状态");
        // 可能触发节能模式或等待中断
    }
}
§任务生命周期

从队列中取出的任务通常会：

  1. 状态从 Ready 转变为 Running
  2. 被加载到 CPU 执行
  3. 根据执行结果重新加入队列或结束
§调度公平性

FIFO 策略保证了调度的公平性：

  • 所有任务都有相同的调度机会
  • 不会出现任务饥饿问题
  • 调度行为完全可预测
§使用示例
// 调度器主循环
loop {
    if let Some(task) = manager.fetch() {
        // 切换到任务执行
        run_task(task);
    } else {
        // 没有任务，等待事件
        wait_for_interrupt();
    }
}
