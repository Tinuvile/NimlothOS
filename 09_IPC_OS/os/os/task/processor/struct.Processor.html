Expand description

处理器状态管理器

管理单个 CPU 核心的执行状态，包括当前正在执行的任务和空闲任务上下文。 作为调度器的核心数据结构，协调任务之间的 CPU 资源分配。

§设计原理

§双上下文模型

处理器维护两种执行上下文：

  • 任务上下文: 用户任务的执行状态（寄存器、栈指针等）
  • 空闲上下文: 调度器的执行状态，用于任务切换时的中转

§所有权管理

通过 Option<Arc<TaskControlBlock>> 管理当前任务：

  • Some(task): CPU 正在执行该任务
  • None: CPU 处于空闲状态，等待新任务调度

§状态转换

处理器状态转换:

   Idle State            Running State
┌─────────────┐         ┌─────────────┐
│   Idle      │ ──────► │   Running   │
│ current=None│Schedule │current=Some │
└─────────────┘         └──────┬──────┘
      ▲                        │
      │  Task Complete/Switch  │
      └────────────────────────┘

§内存布局

Processor 内存结构:
┌─────────────────────────────────────┐
│           Processor                 │
├─────────────────────────────────────┤
│ current: Option<Arc<TCB>>           │
│  └─ Some: Points to Current TCB     │
│  └─ None: Processor Idle            │
├─────────────────────────────────────┤
│ idle_task_cx: TaskContext           │
│  └─ ra: Return Address              │
│  └─ sp: Stack Pointer               │
│  └─ s0-s11: Saved Registers         │
└─────────────────────────────────────┘

§并发安全

Processor 本身不提供线程安全保证，必须通过 UPSafeCell 包装：

  • 单处理器系统中通过禁用中断保证原子性
  • 多处理器系统需要额外的同步机制

§性能特征

  • 内存占用: 约 264 bytes（TaskContext ≈ 264 bytes + Arc 指针）
  • 切换开销: O(1) 常数时间的上下文切换
  • 缓存友好: 紧凑的内存布局提供良好的缓存局部性

Fields§

§current: Option<Arc<TaskControlBlock>>

当前正在执行的任务

  • Some(task): 指向正在 CPU 上执行的任务控制块
  • None: CPU 处于空闲状态，调度器正在寻找下一个可运行任务

使用 Arc 允许任务控制块在调度器和其他组件之间共享所有权。

§idle_task_cx: TaskContext

空闲任务上下文

当没有用户任务运行时，CPU 执行调度器循环的上下文。 包含调度器的寄存器状态，用于在任务切换时保存/恢复调度器状态。

§作用

  • 任务切换中转: 从任务A切换到任务B时的中间状态
  • 调度器状态: 保存调度器循环的执行状态
  • 空闲处理: CPU 空闲时的默认执行上下文

pub fn new() -> Self

创建新的处理器实例

初始化一个空闲状态的处理器，准备接受任务调度。 处理器创建时没有当前任务，空闲上下文被清零。

§Returns

新创建的处理器实例，处于空闲状态

§初始状态
  • current: None - 没有正在执行的任务
  • idle_task_cx: 零初始化的任务上下文
§使用场景
  • 系统初始化时创建全局处理器
  • 多处理器系统中创建额外的处理器核心
§Examples
let processor = Processor::new();
assert!(processor.current().is_none()); // 初始状态为空闲

pub fn current(&self) -> Option<Arc<TaskControlBlock>>

获取当前正在执行的任务（克隆引用）

返回当前任务控制块的克隆引用，保持原始引用不变。 适用于需要访问当前任务信息但不需要取走所有权的场景。

§Returns
  • Some(Arc<TaskControlBlock>) - 当前任务的克隆引用
  • None - 处理器当前处于空闲状态
§引用计数

此函数会增加任务控制块的引用计数：

调用前: Arc::strong_count = n
调用后: Arc::strong_count = n + 1
§使用场景
  • 信息查询: 获取当前任务的 PID、状态等信息
  • 权限检查: 验证当前任务的访问权限
  • 上下文共享: 在多个组件间共享任务引用
§Examples
if let Some(task) = processor.current() {
    println!("Current PID: {}", task.getpid());
    // 任务引用在作用域结束时自动释放
} else {
    println!("No task currently running");
}
§性能考虑
  • 引用计数操作开销很小（原子操作）
  • 适合频繁的只读访问场景
  • 避免不必要的所有权转移

pub fn task_current(&mut self) -> Option<Arc<TaskControlBlock>>

取出当前正在执行的任务（转移所有权）

移除并返回当前任务控制块，将处理器设置为空闲状态。 这是一个所有权转移操作，调用后处理器不再持有任务引用。

§Returns
  • Some(Arc<TaskControlBlock>) - 被取出的任务控制块
  • None - 处理器已经处于空闲状态
§状态变化
调用前: current = Some(task), 处理器运行状态
调用后: current = None,       处理器空闲状态
§使用场景
  • 任务切换: 在调度新任务前取出当前任务
  • 任务完成: 任务退出时清理处理器状态
  • 任务挂起: 将任务转移到等待队列
§调度流程中的作用
1. take_current_task() ──► 取出当前任务
2. 任务状态处理 ────────► 加入就绪队列或退出
3. fetch_task() ───────► 获取新任务
4. 设置新的 current ───► 开始执行新任务
§Examples
// 在任务调度中使用
if let Some(old_task) = processor.task_current() {
    // 处理旧任务（加入就绪队列、退出等）
    handle_task_switch(old_task);
}
// 现在处理器处于空闲状态，可以调度新任务
§注意事项
  • 调用后处理器立即进入空闲状态
  • 必须确保在合适的时机调用，避免任务丢失
  • 通常与任务调度算法配合使用

fn idle_task_cx_ptr(&mut self) -> *mut TaskContext

获取空闲任务上下文的可变指针

返回指向空闲任务上下文的原始指针，用于底层的上下文切换操作。 这是一个私有方法，只在内部上下文切换时使用。

§Returns

*mut TaskContext - 指向空闲上下文的可变原始指针

§安全性考虑

此方法返回原始指针，调用者必须确保：

  • 指针在使用期间保持有效
  • 不会导致数据竞争或内存安全问题
  • 只在适当的同步保护下使用
§使用场景
  • 上下文切换: 在 __switch 汇编函数中使用
  • 调度器循环: 保存/恢复调度器状态
  • 任务切换中转: 作为任务间切换的中介
§上下文切换流程
1. 保存当前任务上下文到任务控制块
2. 恢复空闲上下文 ────────────► 返回调度器
3. 调度器选择新任务
4. 保存空闲上下文 ────────────► 为下次切换准备
5. 恢复新任务上下文 ─────────► 执行新任务
§内存布局

返回的指针指向 idle_task_cx 字段的内存地址， 包含完整的 RISC-V 寄存器上下文信息。
